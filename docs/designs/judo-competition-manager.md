## Problem Statement

Club-level judo competition organizers are stuck between three bad options: federation-grade desktop software (JudoManager) that's overkill, a capable but 2008-era open source desktop app (JudoShiai) with no web registration or mobile support, or an expensive multi-sport SaaS (SmoothComp) that treats judo as one of many sports. The organizer currently duct-tapes together spreadsheets, WhatsApp groups, and paper score sheets to run a tournament. Registration takes days of manual spreadsheet work. Pool generation is error-prone. Scoreboards are either paper or clunky installed software. Results get manually transcribed.

The founder organizes judo events himself and lives this pain directly.

## Demand Evidence

- Founder is the target user, organizes judo competitions and currently uses a mix of spreadsheets, paper, and messaging apps
- JudoShiai on SourceForge has steady downloads but no web/mobile capability, indicating demand for the workflow but frustration with the delivery
- SmoothComp is used by USA Judo for nationals, proving willingness to pay for tournament software at scale
- No modern, web-native, judo-first tool exists at the club/regional level

## Status Quo

Current workflow for a typical club competition:
1. **Registration (days before):** Excel/Google Sheets. Competitors email or WhatsApp their details. Organizer manually enters name, weight class, age, belt, club. Chases missing info. Takes hours.
2. **Pool generation (night before):** Manual grouping by weight class + age division. Draw seeding. Build round-robin pools or repechage brackets in spreadsheets. Print sheets.
3. **Weigh-in (morning of):** Paper checklist. Mark off competitors as they step on the scale. Move no-shows out of pools. Reprint affected brackets.
4. **Competition (day of):** Paper score sheets at each mat. Someone calls out scores. Results walked to a central table. Head referee coordinates verbally.
5. **Results (after):** Manual tabulation. Medal standings computed by hand. Federation results report typed up manually.

Every handoff between steps is manual. Every reprint costs 15 minutes. No live visibility for spectators or coaches.

## Target User & Narrowest Wedge

**Primary user:** The club-level judo competition organizer. Runs 2-10 events per year. Usually also a coach or referee. Not a full-time event manager. Technically comfortable but not a developer. Wants to spend their time coaching, not wrestling with spreadsheets.

**Narrowest wedge:** Registration + automatic pool generation. Competitors register online (name, weight, DOB, belt, club). System auto-assigns to IJF weight categories and age divisions. Generates round-robin pools or single/double repechage brackets per IJF rules. Eliminates 4-8 hours of spreadsheet work per event.

**Second wedge:** Live judo scoreboard. Browser-based, runs on any tablet/TV. Table official controls it, spectators view it on their phones. Ippon, waza-ari, shido, osaekomi timer, golden score, hansoku-make. No software install.

## Constraints

- Must work on spotty gym wifi (scoreboard needs offline resilience or local-network mode)
- IJF rules for weight classes, age divisions, and bracket structures (repechage)
- Multiple concurrent mats (typically 2-4 mats at a club event, up to 8+ at regionals)
- Scoreboard must render cleanly on TV screens (16:9), tablets, and phones
- Registration must handle late entries and weight class changes after weigh-in

## Premises

1. Club-level judo organizers are underserved by current tools
2. The founder is the target user and will dogfood every feature
3. Registration + pool generation is the biggest pre-event pain
4. The scoreboard is the viral feature that spreads word-of-mouth
5. Web-native (no installs, everything in the browser) is the differentiator

## Approaches Considered

### Approach A: Judo-First Web App (Chosen)
Full-stack web app, judo-specific from day one. NestJS + React (proven stack from theraplan). Registration, pool generation (IJF weight classes + repechage), weigh-in confirmation, and browser-based scoreboard.
- Effort: L (human: ~3-4 weeks / CC: ~2-3 days)
- Risk: Low
- Pros: Reuse proven architecture, judo-specific data model, scoreboard as standalone URL
- Cons: Single-sport limits initial market, need offline handling for gyms

### Approach B: Multi-Sport Platform from Day One
Generic competition engine with sport-specific plugins. Judo is first but data model supports any combat sport.
- Effort: XL (human: ~6-8 weeks / CC: ~5-7 days)
- Risk: High (premature abstraction)
- Rejected: "Build for everyone, perfect for nobody." Don't abstract until you know what other sports actually need.

### Approach C: Scoreboard-First MVP
Build only the live scoreboard. No registration, no pools.
- Effort: S (human: ~3-4 days / CC: ~3-4 hours)
- Risk: Low but limited standalone value
- Rejected: A scoreboard alone isn't a business. It's a feature, not a product.

## Recommended Approach

**Approach A: Judo-First Web App.** Build the full registration-to-results flow for judo, using the exact same stack as theraplan (NestJS 11 + Prisma + React 19 + TanStack Query + Tailwind + shadcn/ui). Judo-specific data model, no multi-sport abstraction. Deploy to Fly.io or a VPS.

### Core Data Model

```
Competition
  ├── id, name, date, location, status (DRAFT/REGISTRATION/WEIGH_IN/ACTIVE/COMPLETED)
  ├── organizerId (User)
  ├── mats: Mat[]
  └── categories: Category[]

Category (auto-generated from IJF rules)
  ├── id, name (e.g., "U73 Senior Men")
  ├── weightClass, ageGroup, gender
  ├── bracketType (ROUND_ROBIN | SINGLE_REPECHAGE | DOUBLE_REPECHAGE)
  └── competitors: Competitor[]

Competitor
  ├── id, firstName, lastName, dateOfBirth, weight, belt, club
  ├── categoryId (assigned after weigh-in)
  ├── registrationStatus (REGISTERED/WEIGHED_IN/WITHDRAWN)
  └── matches: Match[]

Mat
  ├── id, number, competitionId
  └── currentMatch: Match?

Match
  ├── id, categoryId, matId, round, poolPosition
  ├── competitor1Id, competitor2Id
  ├── winnerId, winMethod (IPPON/WAZA_ARI/DECISION/HANSOKU_MAKE/FUSEN_GACHI/KIKEN_GACHI)
  ├── scores: { competitor1: ScoreState, competitor2: ScoreState }
  ├── status (SCHEDULED/ACTIVE/COMPLETED)
  └── duration, goldenScore: boolean

ScoreState (JSON)
  ├── wazaAri: number (0-2, 2 = ippon equivalent)
  ├── shido: number (0-3, 3 = hansoku-make)
  └── osaekomi: number (seconds, 10 = waza-ari, 20 = ippon)
```

### Authentication & Access

- **Organizer:** email/password account. Creates and manages competitions.
- **Competitor:** no account needed for v1. Registers via public link with name + email. Receives a confirmation email with a link to view their draw/schedule.
- **Table Official:** organizer generates a per-mat PIN code (4-6 digits). Table official enters the PIN at `/mat/{id}/control` to gain scoreboard control. No account needed. PIN displayed to organizer only. If someone enters the wrong PIN, they can only view, not control.

### Bracket Rules (IJF)

- **Pool size thresholds:** 1-4 competitors in a category = round-robin pool. 5+ competitors = single repechage bracket. 8+ competitors = double repechage.
- **Seeding:** competitors ranked by belt grade within each category. No seeding for v1 (random draw). Federation seeding is a v2 feature.
- **Two waza-ari = ippon:** match terminates immediately. The ScoreState tracks waza-ari count; the match engine auto-terminates at 2 and records the win method as WAZA_ARI.
- **Three shido = hansoku-make:** match terminates, opponent wins.
- **Osaekomi:** 10s = waza-ari, 20s = ippon. Timer runs independently on the control view. If osaekomi reaches 20s, match auto-terminates.
- **Match duration:** configurable per age group. Defaults: Senior 4 min, Junior 4 min, Cadet 4 min, U13 3 min. Organizer can override per competition.
- **Golden score:** triggered when regulation time ends with no score difference. First score wins. No time limit. Shido in golden score = hansoku-make (opponent wins).
- **Round-robin tiebreakers:** direct confrontation result, then number of ippons, then number of waza-ari wins, then least shidos received. If still tied, a golden score playoff.
- **Additional win methods:** KIKEN_GACHI (withdrawal during match) added to the Match model enum.

### Weigh-in Rules

- Competitors weigh in at their registered category.
- **Over weight:** organizer can move the competitor up one weight class if space allows, or mark them as withdrawn.
- **Under weight:** stays in registered category (no advantage rule in IJF).
- **No-show:** marked as withdrawn. Affected pool auto-regenerates (byes adjusted).
- **Second weigh-in:** not supported in v1. Competitor either makes weight or moves up/withdraws.

### Notification

- v1: email only. Competitor gets a confirmation email on registration with a link to check their draw and schedule. No push notifications, no SMS.
- Spectators check the public URL. No notifications.

### v1 Feature Scope

**Pre-event (organizer):**
- Create competition with date, location, rules config
- Online registration form (public link, no login required for competitors)
- Auto-categorize by IJF weight class + age division
- View/edit registrations, handle late entries
- Generate pools/brackets per category (round-robin for small pools, repechage for larger)
- Assign categories to mats

**Day-of (organizer + table official):**
- Weigh-in checklist (mark competitors as weighed, flag over/under weight)
- Move/withdraw competitors, regenerate affected pools
- Match sequencing per mat (which fight is next)
- Table official scoreboard controls (dedicated tablet interface)

**Live (everyone):**
- Public scoreboard URL per mat (TV/projector display)
- Spectator view showing all mats, current fights, upcoming schedule
- Real-time results as matches complete

**Post-event:**
- Medal standings per category (rendered on screen, no PDF export in v1)
- Results visible on the public competition URL

### Tech Stack

- **Backend:** NestJS 11 + Prisma + PostgreSQL (same as theraplan)
- **Frontend:** React 19 + Vite + TanStack Query + Tailwind + shadcn/ui
- **Real-time:** Socket.IO for scoreboard sync (built-in reconnection, room-per-mat)
- **Deploy:** Fly.io initially, VPS later (same pattern as theraplan)

### Scoreboard Architecture

The scoreboard is the most technically interesting piece. Three views of the same data:

1. **Control view** (table official tablet) — buttons for ippon, waza-ari, shido, start/stop osaekomi, golden score toggle. Touch-optimized, large tap targets.
2. **Display view** (TV/projector) — the classic judo scoreboard. Two competitors, scores, timer, shidos. Full-screen, high contrast, readable from 20 meters.
3. **Spectator view** (phone) — compact version showing current fight + upcoming schedule for each mat.

All three connected via WebSocket to the same match state. Table official pushes updates, display and spectator views receive them in real-time.

**Real-time sync:** Use Socket.IO (reconnection handling built-in, room-per-mat). The control view is the single source of truth for a match. It pushes score events (not full state) to the server, which broadcasts to display and spectator views.

**Osaekomi timer:** The control view starts/stops osaekomi and sends the event + timestamp. Display and spectator views run their own local timers started from the received event, corrected on each heartbeat. This prevents network jitter from making the timer jump.

**Offline resilience:** The control view queues score events in IndexedDB when disconnected. On reconnect, it replays the event queue in order. The server applies events idempotently (each event has a monotonic sequence number per match). Display views show "Connection lost" overlay during disconnect but the table official's controls never freeze. The match timer is client-side on the control view, authoritative, and synced on reconnect.

## Open Questions

1. **Competitor accounts:** Do competitors need to create accounts, or is anonymous registration (name + email) sufficient for v1?
2. **Federation integration:** Do you need to export results in a specific federation format (e.g., for ranking points)?
3. **Payment:** Should registration include online payment, or is that handled separately (cash at the door)?
4. **Multi-day events:** Is v1 single-day only, or do you need multi-day support?

## Success Criteria

- Successfully run one real judo competition using the software (your own event)
- Registration takes < 5 minutes per competitor (vs hours of spreadsheet work)
- Pool generation is automatic and correct per IJF rules
- Scoreboard works on a tablet + TV without freezing or losing state
- At least one other organizer asks to use it for their event

## Distribution Plan

- Web app deployed to Fly.io or VPS (same pattern as theraplan)
- Organizers access via URL (no install)
- Competitors register via public link (no account needed for v1)
- Scoreboard display via URL (cast to TV via Chromecast/HDMI laptop)
- GitHub repo (consider open source to attract judo community contributions)
